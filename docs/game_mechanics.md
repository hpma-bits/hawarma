# 烹饪游戏机制总结

## 一、游戏概述

这是一款烹饪模拟游戏，玩家需要根据客户订单准备食材、烹饪菜肴并上菜。自动化bot通过图像识别和UI自动化技术，实现了对游戏的半自动/自动化控制。

### 2.4 订单系统

- **订单显示区域**：最多4个并发订单显示位置
- **订单提交位置**（Pickup Station）：菜品完成后送往的位置，共4个，与订单显示位置一一对应
- **订单类型**：普通订单和加急订单（底色为红色，**得分更高，时限更短**）
- **优先级**：加急订单(rush order)具有更高的优先级，应该优先处理
- **超时机制**：订单超时后slot清除
- **订单显示区域补位机制**：
  - 订单提交后，右侧（后面）的订单会自动向左补位
  - 后续订单的提交位置（pickup station）也随之向左移动
  - 新的订单刷新在最左边的空显示位置
  - 如果4个订单显示位置都满了，则不会刷新新订单，直到有订单被完成或超时
  - **补位动画时间**：订单提交后的补位与新订单刷新需要动画时间，约1秒（建议保守1.5秒），此期间scan操作可能捕捉到未刷新完的页面，应避免在提交后马上进行scan
- **订单阶段**：
  1. PENDING - 待处理
  2. HEATING - 加热中
  3. OFF_HEAT - 加热完成
  4. READY_TO_SEASON - 等待调味
  5. SEASONING - 调味中
  6. SERVING - 上菜中
  7. COMPLETED/FAILED - 完成/失败

## 三、游戏流程

### 3.1 订单处理规则

- **优先级规则**：加急订单(rush order)具有更高的优先级，应该优先处理
- **UI操作序列化**：同一时刻只能有1个UI操作执行，所有点击、拖拽等操作必须串行
- **订单显示区域补位**：订单提交后，后续订单自动向左补位，新订单从最左边刷新

### 3.2 订单处理管道

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  订单检测   │ →  │  食材准备   │ →  │   烹饪     │ →  │  调味上菜   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
      ↓                  ↓                  ↓                  ↓
 DetectionService  CookingService   CookingService   CookingService
```

### 3.3 订单检测

1. 截图获取游戏画面
2. 在4个订单区域识别订单
3. 通过第一个食材+厨具组合识别配方：
   - 每个订单的第一个食材与厨具的组合是独一无二的
   - 厨具图标(cooker-icon)位于订单区域的**最左边1/4处**，第一个食材对应的厨具位于此处 # 准确的说是第一个食材对应的厨具位于最左边1/4处
   - 先检测第一个食材本身（ROI为订单区域的最左边1/4）
   - 如果没有冲突（只有一个配方使用该食材），则直接确定菜谱
   - 如果有冲突（多个配方使用相同食材），则再检测厨具来区分
4. 检测是否为加急订单（底色为红色）
5. 检测调料偏好
6. 使用timer-icon检测游戏是否开始

**检测阈值**：
- 配方检测：0.8
- 食材匹配：0.7
- 厨具匹配：0.7

**检测区域说明**：
- 订单区域被等分为4个ingredient slots，每个slot占1/4宽度
- Slot 0（最左边）：第一个食材 + 第一个食材对应的厨具图标
- Slot 1-3：其他食材位置

### 3.4 食材准备策略

- **优先使用库存**：已有预烹饪食材时直接使用
- **缺失时从头烹饪**：库存不足则开始新的烹饪任务

### 3.5 库存管理

- **stockpile area分配**：3个stockpile area可以动态调整，理想状态是同时分析多个订单需要的食材，利用空闲厨具烹饪后暂存到stockpile区
- **库存上限**：每个食材最多5份
- **动态策略**：
  - 分析当前所有待处理订单
  - 优先烹饪后续订单需要的食材
  - 不需要囤积大量食材，按需动态调整
- **优先级评分**基于三个因素：
  1. 食材使用频率
  2. 烹饪设备竞争程度（使用频繁的烹饪设备优先）
  3. 烹饪时间（时间越长越值得囤积）
- **Cooker留存时间**：食材烹饪完成后在cooker上最多只能留存**5秒**，超过则需要移到厨余垃圾箱
- **厨余垃圾箱位置**：(130, 560)
- **Stockpile配合优化**：
  - 当多个订单并发处理时，可以将后续订单的食材先烹饪并存放到stockpile区
  - 当前一个订单提交完成后，立即从stockpile区取出食材组装后续订单
  - 注意：stockpile区可能已满（存储了其他食材），需合理规划

### 3.6 烹饪执行

1. 将食材拖入对应烹饪设备
2. 等待烹饪完成
3. 将食材拖到组装台（**组装台一次只能处理一个订单**）
4. 逐次添加食材和调味品
5. 送到取餐区

## 四、并发机制

### 4.1 异步架构

使用Python `asyncio`实现并发处理：

- **全局UI操作锁**：同一时刻只能有1个UI操作执行，确保操作序列化
- **asyncio.Lock**：保护共享资源
  - 订单槽位锁
  - 烤箱锁（每个烤箱独立锁）
  - 组装台锁
  - 库存区锁
- **asyncio.create_task**：创建后台任务
- **asyncio.gather**：并发执行多任务（内部仍通过UI锁串行执行）

### 4.2 后台任务

- **订单扫描任务**：持续检测新订单
- **库存管理任务**：主动烹饪高优先级食材
- **订单处理管道**：协调4个订单的并发处理

## 七、数据模型

### 7.1 Recipe（配方）

```python
class Recipe(BaseModel):
    slug: str              # 唯一标识
    name: str              # 显示名称
    raw_ingredients: list[str]  # 所需食材
    cookers: list[str]          # 对应烹饪设备（用于烹饪）
    cookers_layout: list[str]   # 布局用烹饪设备（用于界面显示）
    cook_durations: list[float] # 烹饪时长
    condiments: list[str]       # 可选调料
```

### 7.2 Order（订单）

```python
@dataclass
class Order:
    recipe: Recipe              # 配方
    is_rush: bool               # 是否加急
    condiment_preference: dict  # 调料偏好 {condiment_name: count}
    order_id: int               # 订单唯一ID（自动递增）
    done: bool                  # 是否已完成
    current_stage: OrderStage   # 当前阶段
    ingredient_prep_task: Task  # 食材准备任务
    finish_order_task: Task     # 订单完成任务（调味+上菜）
    served_ts: float            # 上菜时间戳
```

## 九、总结

该烹饪游戏的核心机制是：

1. **多订单并发处理** - 最多同时处理4个客户订单
2. **库存预烹饪** - 提前准备常用食材以减少等待时间
3. **设备调度** - 合理分配4种烹饪设备
4. **流水线作业** - 订单依次经过检测→准备→烹饪→调味→上菜

自动化bot通过图像识别自动检测订单，通过异步并发最大化烹饪效率，通过库存管理减少空等时间，从而实现高效率的订单完成。
