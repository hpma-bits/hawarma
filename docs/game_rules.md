# 游戏规则

> 本文档定义游戏的核心规则，作为模拟环境和验证的唯一依据。
> 不涉及自动化实现细节。

---

## 1. 游戏界面元素

```
┌────────────────────────────────────────────────────┐
│  订单区 (4 slot)                                    │
│  [slot0] [slot1] [slot2] [slot3]                   │
│                                                    │
│  灶台区 (4)          组装站 (1)                     │
│  [grill]             [assembly]                    │
│  [oven]                                          │
│  [skillet]                                         │
│  [pot]                                             │

│                                                    │

```

## 2. 坐标确定规则

### 2.1 概述

游戏界面中各元素的位置由玩家选择的菜谱顺序动态决定，而非固定映射。主要包括三个区域：灶台区(Cookers)、食材区(Ingredients)和调料区(Condiments)。

### 2.2 灶台区(Cookers)

- **槽位数量**：游戏最多有4个灶台槽位，但并非每局都解锁全部4个
- **槽位分配规则**：
  - 需要1种cooker → 占据索引 `[1]`
  - 需要2种cooker → 占据索引 `[1, 2]`
  - 需要3种cooker → 占据索引 `[0, 1, 2]`
  - 需要4种cooker → 占据索引 `[0, 1, 2, 3]`
- **顺序确定**：按菜谱选择顺序，将各菜谱的`cookers_layout`元素去重后依次排列
- **示例**：
  ```
  菜谱1: cookers_layout = ["oven", "pot"]
  菜谱2: cookers_layout = ["skillet"]
  菜谱3: cookers_layout = ["grill", "pot"]
  
  去重后顺序: ["oven", "pot", "skillet", "grill"]
  共4种cooker → 占据 [0, 1, 2, 3]
  ```

### 2.3 调料区(Condiments)

- **槽位索引**：从下到上、从左到右排列（索引0-7）
- **顺序确定**：按菜谱选择顺序，将各菜谱的`condiments`元素去重后依次排列
- **示例**：
  ```
  菜谱1: condiments = ["buttermilk_cream", "midsummer_onion"]
  菜谱2: condiments = ["hearthspice", "acacia_honey"]
  菜谱3: condiments = ["hearthspice", "fleur_de_sel"]
  
  去重后顺序: ["buttermilk_cream", "midsummer_onion", "hearthspice", "acacia_honey", "fleur_de_sel"]
  分配索引: [0, 1, 2, 3, 4]
  ```

### 2.4 食材区(Ingredients)

- **槽位索引**：从下到上、从左到右排列
- **顺序确定**：按菜谱选择顺序，将各菜谱的`raw_ingredients`元素去重后依次排列，然后**反转顺序**再分配索引
- **示例**：
  ```
  菜谱1: raw_ingredients = ["clearwater_fish", "creamfield_rice"]
  菜谱2: raw_ingredients = ["dough_wrappers", "tender_lamb"]
  菜谱3: raw_ingredients = ["prime_cut_beef", "vining_marjoram"]
  
  去重后顺序: ["clearwater_fish", "creamfield_rice", "dough_wrappers", "tender_lamb", "prime_cut_beef", "vining_marjoram"]
  反转后分配: 
    vining_marjoram → 索引0
    prime_cut_beef → 索引1
    tender_lamb → 索引2
    dough_wrappers → 索引3
    creamfield_rice → 索引4
    clearwater_fish → 索引5
  ```

### 2.5 坐标配置来源

所有位置坐标定义在 `configs/config.yaml` 中：
- `raw_ingredients_positions`: 食材区位置列表
- `cookers_positions`: 灶台区位置列表  
- `condiments_positions`: 调料区位置列表

## 3. 订单

### 3.1 订单属性

- **配方**：决定需要哪些食材、用哪个灶台、烹饪多久
- **调料偏好**：每种调料需要几份（1 或 2）
- **是否 rush**：rush 订单时限更短、得分更高
  - Rush 订单背景为**红色**，普通订单为其他颜色
  - 可通过检查订单卡特定位置像素颜色来识别
  - 红色在订单生命周期内保持一致

### 3.2 配方约束

- **食材唯一性**：每种食材在同一份菜品中最多出现 1 份，不存在需要 2 份相同食材的菜谱
- **食材数量**：一份菜品最多需要 2 种不同食材（各 1 份），部分菜谱仅需 1 种食材
- **食材-厨具绑定**：`raw_ingredients` 与 `cookers` 按位置一一对应，即 `raw_ingredients[i]` 必须使用 `cookers[i]` 烹饪。例如：
  - `gildedShoreRisotto`: `["clearwater_fish", "creamfield_rice"]` → `["oven", "pot"]`，表示 clearwater_fish 需用 oven 烹饪
  - `braisedNewYearFish`: `["clearwater_fish"]` → `["skillet"]`，表示 clearwater_fish 需用 skillet 烹饪
  - 该组合决定唯一烹饪时长
- **配方唯一性**：两个不同的 recipe 不能包含相同的 ingredient-cooker 组合但不同的 duration

### 3.3 订单显示

- 最多 4 个订单同时显示（slot 0-3）
- 新订单出现在最左侧空 slot
- 如果 4 个 slot 都满了，新订单不会出现
- **订单生成间隔**：游戏开始后，每隔 **3-5 秒随机**自动刷新一个新订单，平均约 4 秒
  - 实际间隔是随机的，不是固定的 4 秒
  - 第1个订单：游戏开始后约 3-5 秒出现
  - 后续订单：从上次刷新时刻开始，随机等待 3-5 秒
  - 直到游戏结束或所有槽位已满
- **Rush 订单识别**：Rush 订单的订单卡背景为红色（普通订单为其他颜色），可通过检查订单卡特定位置的像素颜色来识别

### 3.4 Slot 位移与订单刷新

订单被提交后：
1. 该 slot 变为 None
2. 右侧所有订单左移填补空位
3. 位移需要约 1.5 秒动画时间
4. **状态立即更新**：内部状态在提交后立即更新（slot 0 立即包含原 slot 1 的订单），动画只是视觉效果
5. 动画期间限制：
   - 不能扫描订单区（会捕获未刷新的屏幕）
   - 不能送餐提交（assembly → pickup）
   - 可以进行其他操作：烹饪、调味、移到组装站、清理灶台等

**订单刷新规则（关键补充）**：
1. **提交后立即刷新**：如果订单被提交后，场上**没有订单**，系统会**立即刷新一个新订单**（不等待间隔）
2. **新订单动画时间**：新出现的订单有 **1 秒动画时间**，动画结束后（1秒后）玩家才能检测到该订单
3. **提交后部分订单时的刷新**：如果场上有 4 个订单，提交了 1 个后剩下 3 个，则从**提交时刻开始计时**，**随机等待 3-5 秒后**刷新新订单
4. **过期后刷新**：每次订单过期都**重置**计时器，从该时刻开始重新计时，随机等待 3-5 秒
5. **间隔是随机的**：每次刷新计时都是独立随机事件，范围 3-5 秒，平均 4 秒

示例：
```
场景1 - 提交后无订单，立即刷新：
提交前: [A]                slots: [0]  (只有1个订单)
提交 A: [_]  → 位移 → [B]  slots: [0]  (立即刷新B，1秒后可见)

场景2 - 提交后仍有订单，按随机间隔：
提交前: [A, B, C, D]      slots: [0, 1, 2, 3]  (4个订单)
提交 A: [B, C, D, _]      slots: [0, 1, 2]    (剩3个)
随机3-5秒后: [B, C, D, E]  slots: [0, 1, 2, 3]  (新订单E，间隔随机)

场景3 - 多个订单连续过期，每次过期重置计时器（随机间隔）：
t=44: [A, B, C, D]        (4个订单)
t=44: A 超时 → [B, C, D, _]   (剩3个，开始随机3-5秒计时)
t=48: 新订单 E 出现 → [B, C, D, E]  (44+4=48，假设随机到4秒)
t=48: B 超时 → [C, D, E, _]   (剩3个，重置计时器，重新开始随机3-5秒计时)
t=52: 新订单 F 出现 → [C, D, E, F]  (48+4=52，假设随机到4秒)
```

### 3.5 订单超时

- **时限与配方相关**：超时时限取决于订单所需食材的烹饪时长
  - **普通订单**：约 55-75 秒（与 recipe 的食材耗时相关）
  - **Rush 订单**：约 30-45 秒（比普通订单短约 40%）
  - 具体时长由游戏根据当前选中的菜谱动态决定
- 超时后订单失败，触发 slot 位移（同订单提交）
- **超时不影响 assembly station**：已放置的食材保留，可用于其他需要该食材的订单
- 只有符合 recipe 要求的食材组合才能成功提交，否则会被退回 assembly station
- **超时后的刷新规则**：
  - 每次订单过期都**重置**随机计时器（3-5秒），从该时刻开始重新计时
  - 示例：t=44 订单过期 → 随机3-5秒后（如t=48）生成新订单；t=48 另一订单过期 → 重置计时器，随机3-5秒后生成新订单

### 3.6 游戏时间限制

- **总游戏时长**：**90-110 秒**（根据玩家等级/进度不同而变化）
  - 新手玩家：约 90 秒
  - 高级玩家：可达 110 秒
  - 游戏开始前由玩家在配置中设定
  - 游戏从 0 秒开始，到设定的时长结束
  - 时间到时游戏自动结束
  - 游戏结束后不再生成新订单
- **游戏目标**：在限定时间内尽可能完成更多订单，获得更高分数

### 3.7 得分机制

- **基础得分**：完成订单获得基础分数
- **调料匹配**：添加的 condiment 与订单偏好匹配可获得更高分数
  - 必须添加 recipe 要求的 condiment 类型和数量
  - 无效 condiment（非 recipe 要求）不影响得分但也不加分
  - 调料总数量上限为 3 份，超过部分被忽略

#### Visibility 加成（已完成订单总分加成）

订单的基础分数会随着**已完成订单的总 visibility** 的提高而获得额外加成。每完成一个订单，该订单的 visibility 值会被累加到总 visibility 中，后续订单的得分根据当前总 visibility 所在区间享受对应的加成比例。

**Visibility 区间与加成比例**：

| 总 Visibility 区间 | Normal 订单加成 | Rush 订单加成 |
|:------------------:|:---------------:|:-------------:|
| [0, 40)            | +0%             | +60%          |
| [40, 80)           | +10%            | +100%         |
| [80, 160)          | +20%            | +150%         |
| [160, 240)         | +30%            | +200%         |
| [240, 360)         | +40%            | +250%         |
| [360, ∞)           | +50%            | +300%         |

**计算方式**：
```
订单得分 = (base points + visibility) × (1 + 加成比例)
```

**关键规则**：
- 订单的加成比例在**订单生成（刷新）的瞬间**即锁定，由当时的总 visibility 决定
- 该订单自身的 visibility 在完成之后才累加到总 visibility 中，因此不影响本订单的得分，但会影响之后生成的新订单
- 不同 recipe 的 visibility 值不同，可在 `data/reward.csv` 中查看
- **最终总分** = 所有已完成订单的得分之和 + 游戏结束时的总 visibility

## 4. 烹饪

### 4.1 流程

1. 将食材从食材区 swipe 到对应灶台
2. 等待烹饪完成（时间由配方决定）
3. 将成品从灶台 swipe 到组装站或库存区

### 4.2 灶台约束

- 每个灶台同时只能处理一个食材
- **并行烹饪支持**：多个灶台可以同时独立工作，各自维护自己的计时器
  - 例如：grill 烹饪牛肉（3秒），同时 oven 烹饪鱼（4秒）
  - 两个灶台完全独立，互不干扰
- 烹饪完成后，食材在灶台最多停留 **5 秒**
- 超过 5 秒未移走 → 食材过期（糊掉），必须 swipe 到垃圾桶 (130, 560)
- 灶台在食材被移走后才释放
- **可烹饪任意食材**：即使当前没有订单需要该食材，也可以烹饪。食材会一直留在灶台直到过期或被移走
- **烹饪时长**：根据 ingredient-cooker 组合从配方数据中查找，每个合法组合有唯一且固定的烹饪时长

## 5. 库存

### 5.1 机制

- 3 个库存 slot，每种食材最多存 **5 份**
- 空闲时可将烹饪好的食材存入库存（swipe 灶台 → 库存 slot）
- 需要时可直接从库存 swipe 到组装站，跳过烹饪

### 5.2 库存约束

- **厨具绑定**：每个库存 slot 只能存放由**同一种厨具**烹饪的**同一种食材**
- **数量上限**：每个 slot 最多存 5 份
- **兼容性规则**：
  - 同种食材 + 不同厨具 = 不能共存于同一 slot
  - 不同食材 + 同种厨具 = 不能共存于同一 slot
- **失败处理**：尝试将不兼容的食材移入已有食材的 slot 时，操作失败，食材留在原处

## 6. 组装站

### 6.1 约束

- **同一时刻只能存放一份菜品**：组装站上最多存放一个完整 recipe 所需的食材组合
- **食材兼容性**：组装站上的食材必须属于同一个 recipe。如果尝试添加不同 recipe 的食材，该食材会被退回来源位置
- **自动组装**：当组装站集齐某个 recipe 所需的所有食材后，菜品自动完成组装，此时可以添加 condiment 或直接提交

### 6.2 流程

1. 食材逐一到达组装站
2. 所有食材到齐 → 进入调味阶段
3. 调料逐一从调料区 swipe 到组装站（双份 swipe 两次）
4. 全部完成后 swipe 组装站 → 取餐台

### 6.3 约束

- **食材上限**：组装站上每种食材最多 1 份，最多 2 种不同食材（单食材菜谱只需 1 种）
- **调料上限**：一份菜品最多 3 份调料
- **无效调料**：非 recipe 要求的调料 swipe 到组装站不会产生任何效果（菜品状态不变）
- **食材不绑定订单**：组装站上的食材不属于特定订单，可被任意需要该 recipe 的订单使用
- **失败退回**：尝试向组装站添加不兼容的食材时，该食材会被退回来源位置，不进入组装站
- **食材视觉**：玩家可以区分组装站上**不同种类**的食材（如区分牛肉和鸡蛋），但**无法区分同种食材用不同厨具烹饪**（如无法区分烤牛肉和煮牛肉）

## 7. 取餐台

- 4 个取餐台，与订单 slot 一一对应
- 组装站 → 取餐台[N] 的 swipe 表示将订单送到第 N 个取餐台
- 送餐后订单完成，slot 位移

## 8. 操作约束

| 规则 | 说明 |
|------|------|
| 每次只能执行一个 UI 操作 | swipe/click 必须串行 |
| 动画窗口限制 | slot 位移动画期间（~1.5s）：禁扫描订单区、禁送餐提交；**原因：UI 在变化，扫描得到的图片可能马上就变化，送餐提交同理**；其他操作不受限 |
| swipe 坐标必须合法 | 起点/终点必须是游戏元素位置 |
| 灶台过期清理 | 食材在灶台最多停留 5 秒 |
| 丢弃食材 | 可将 cooker、stockpile、assembly station 的食材丢弃到 trash |

## 9. 操作类型

### 9.1 Gastronome 模式操作

| 操作 | 起点 | 终点 | 含义 |
|------|------|------|------|
| 烹饪 | 食材区[ing] | 灶台[cooker] | 开始烹饪 |
| 移到组装站 | 灶台[cooker] | 组装站 | 烹饪完成，送组装 |
| 移到库存 | 灶台[cooker] | 库存区[slot] | 烹饪完成，存库存 |
| 从库存取出 | 库存区[slot] | 组装站 | 跳过烹饪 |
| 调味 | 调料区[cond] | 组装站 | 添加调料 |
| 送餐 | 组装站 | 取餐台[slot] | 提交订单 |
| 丢弃 | 灶台/库存/组装站[食材] | 垃圾桶 | 主动丢弃任意位置的食材 |
| 清理 | 灶台[cooker] | 垃圾桶 | 清理过期食材（等同丢弃） |

### 9.2 Dessert 模式操作

| 操作 | 起点 | 终点 | 含义 |
|------|------|------|------|
| 添加食材 | 食材区[ing] | 搅拌盆 | 将食材放入搅拌盆 |
| 调味 | 调料区[cond] | 搅拌盆 | 在搅拌盆中添加调料 |
| 搅拌 | 搅拌盆 | 搅拌盆 | 在搅拌盆内往复 swipe |
| 移到灶台 | 搅拌盆 | 灶台[cooker] | 搅拌完成后送入灶台烹饪 |
| 送餐 | 灶台[cooker] | 取餐台[slot] | 烹饪完成后直接从灶台出餐 |
| 清空搅拌盆 | 搅拌盆 | 垃圾桶 | 清空搅拌盆中的食材 |

## 10. 甜点模式

### 10.1 概述

甜点模式是游戏的另一种模式，与 Gastronome 模式完全独立。两种模式不会出现在同一局游戏中。

### 10.2 甜点流程

```
食材区 → 搅拌盆 → 调味 → 搅拌 → 灶台烹饪 → 取餐台
```

详细步骤：
1. 食材 A → 搅拌盆
2. 食材 B → 搅拌盆
3. 调味品 → 搅拌盆（依次添加）
4. 搅拌（从搅拌盆坐标按下，向左滑动 400px，持续 1.5s）
5. 搅拌盆 → 灶台（直接滑动到 cooker）
6. 烹饪完成 → 取餐台（直接从灶台出餐）

### 10.3 甜点约束

- **双食材**：所有甜点都是双食材
- **订单检测**：只需检测第一个食材即可确定菜谱（第一个食材唯一确定）
- **无组装站**：甜点不使用组装站，使用搅拌盆
- **直接出餐**：烹饪完成后直接从灶台出餐，不经过组装站
- **超时机制**：甜点有超时机制，normal 70s+，rush 40s+
- **烹饪后出餐**：烹饪完成后只有 5 秒时间去 serve，超过 5 秒后就要清空 cooker

### 10.5 甜点订单特点

- **订单检测**：只需检测第一个食材即可确定菜谱
- **第一个食材唯一确定**：第一个食材可以唯一确定菜谱
- **特殊变种**：mooncake 和 snowskin mooncake 有特殊变种
- **冲突处理**：如果出现相同第一个食材的菜谱，需要检查第二个食材

### 10.6 甜点并发处理

- **流水线决策**：当当前订单在烹饪时，开始下一个订单的食材收集和搅拌
- **并发订单**：一般同时处理 2 个订单
- **优先级**：rush 订单优先处理，然后先进先出
- **资源管理**：
  - 搅拌盆：一次只能处理一个订单
  - cooker：一次只能烹饪一个甜点
  - 库存：一次只能存储一个半成品

### 10.7 甜点半成品库存

- **储存位置**：(675, 860)
- **储存容量**：只能储存一份半成品
- **储存时机**：搅拌完成后可以储存
- **取出时机**：当下一个订单需要相同菜谱时取出
- **无超时**：半成品库存没有超时机制

### 10.8 甜点错误处理

- **操作失败**：记录日志，假设操作成功
- **边界情况**：
  - 搅拌盆已满：记录日志，返回失败
  - cooker 已满：记录日志，返回失败
  - 库存已满：记录日志，返回失败
  - 烹饪超时：清空 cooker

### 10.9 甜点监控和日志

- **监控指标**：
  - 订单处理时间
  - 资源使用情况
  - 成功率
- **日志记录**：
  - 详细的操作日志
  - 错误和异常日志
  - 使用现有的日志级别设置
